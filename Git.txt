Git es un sistema de control de versiones distribuido.

💬️ Que es no distribuido?
que todo dependia de único sitio, si ese sitio se borraba el código se veia afectado. 

💬️ Que es distribuido?
que cada programador puede tener una copia del código, si que se afecte el código.

💬️ Sistema de control de versiones?
Llevar un historial de todos los cambios que haya en nuestro proyecto o documentar los cambios.

💬️ Git remote
- git branch -m 
- git branch -M 

	-m (minúscula): Es un renombrado seguro. Si intentas renombrar una rama a un nombre que ya existe, Git te detendrá y te dirá: "Oye, ya existe una rama llamada así, no quiero sobrescribirla".

	-M (mayúscula): Es un renombrado forzado. Le dice a Git: "No me importa si ya existe otra rama con ese nombre, bórrala y ponle este nombre a la rama actual". Es el modo "fuerza bruta".
	
- git remote add origin https://github.com/NyoEvil/hello-git.git (conectarnos remotamente con github)
- git push -u origin main (mandar todo el proyecto a github)

    - push: Sube tus commits al servidor.

    -u (set-upstream): Esta es la parte inteligente. Le dice a Git: "Recuerda que, de ahora en adelante, mi rama main local está unida a la rama main del remoto bon".

        ¿El beneficio? La próxima vez que quieras subir código, ya no tendrás que escribir todo el comando; con un simple git push a secas, Git ya sabrá que debe ir a bon main.

    origin: El nombre del remoto que creaste antes.
    main: La rama que estás subiendo.

💬️ Git Cherry-Pick y Rebase
1. Git Rebase (La Máquina del Tiempo)

Imagina que mientras tú trabajabas en tu rama login, alguien más (o tú mismo desde tu otra laptop) subió cambios a main. Tu rama login ahora está "desactualizada".

- git rebase main: Toma todos tus commits de login, los "despega" momentáneamente, actualiza tu rama con lo nuevo de main y luego vuelve a pegar tus commits encima.

        Resultado: Un historial limpio y lineal, como si nunca te hubieras quedado atrás.

- git rebase -i (Interactivo): Es el "Modo Dios". Te abre una lista de tus últimos commits y te permite:

        Fusionar dos commits en uno solo (Squash).
        Cambiar el nombre de un mensaje de commit viejo.
        Eliminar un commit que ya no quieres.
        Se usa así: git rebase -i HEAD~3 (para editar los últimos 3 commits).

¿Qué hacer si algo sale mal durante el rebase?

	- git rebase --abort: "¡Pánico! Me equivoqué, vuelve todo a como estaba antes de empezar el rebase".

	- git rebase --continue: "Ya arreglé el conflicto de código, sigue pegando los commits que faltan".

2. Git Cherry-pick (Cosechando Frutas)

Imagina que en una rama de pruebas hiciste 10 commits, pero solo uno de ellos (el que tiene el código de los sockets) es el que realmente sirve y lo quieres en tu rama main.

    - git cherry-pick 34243k: Vas a la otra rama, buscas el ID del commit (el hash como 34243k) y lo "copias y pegas" en tu rama actual.

        ¿Qué se le pasa? El ID o Hash del commit que quieres traer.

Los comandos de control:

    - git cherry-pick --continue: Si hubo un conflicto al traer el commit, lo arreglas y usas esto para terminar de "pegarlo".

    - git cherry-pick --abort: "Mejor no traigas nada, cancela el copiado".

Comando	          ¿Cuándo usarlo?	       Lo que hace
Rebase	Quieres actualizar tu rama completa. Mueve toda tu rama encima de otra.
Cherry-pick	Solo quieres UN commit específico.	Copia un cambio puntual de un sitio a otro.

Un consejo: 

El rebase es una herramienta potente pero peligrosa. Regla de oro: Nunca hagas rebase de una rama que ya subiste a GitHub y que otros están usando, porque les cambiarás el historial y causarás un caos. Úsalo solo para limpiar tus ramas locales antes de hacer el push final.

# ejemplo:

1. Git Rebase (El Actualizador)

El Escenario: Estás en tu laptop en la rama desarrollo-sockets. Mientras programabas, en GitHub alguien (o tú mismo desde la otra laptop) subió cambios a main. Tu rama se quedó "atrás".
Ejemplo de git rebase main:

    Estás en tu rama: git checkout desarrollo-sockets

    Te traes lo nuevo: git fetch bon

    Ejecutas el rebase: git rebase bon/main

    ¿Qué pasó? Git quitó tus commits un momento, puso los nuevos de main y luego volvió a poner los tuyos al final. Ahora tu historia es una línea recta perfecta.

Ejemplo de git rebase -i HEAD~3 (El Limpiador):

Tienes estos commits feos:

    work: avance 1

    work: avance 2

    fix: error de dedo

Quieres que se vea como uno solo profesional.

    Escribes: git rebase -i HEAD~3

    Se abre un editor de texto. Verás la palabra pick al lado de cada commit.

    Cambias pick por squash (o solo s) en los dos últimos.

    Guardas y cierras.

    Resultado: Git fusiona los 3 en uno solo. Ahora puedes ponerle un nombre épico: feat: sistema de sockets terminado.
    
    Al lado de cada commit verás una palabra (por defecto es pick). Puedes cambiarla por:fsdfa

    pick: Deja el commit tal cual.

    reword: Úsalo si escribiste mal el mensaje del commit (ej. "commit progrema" -> "commit programa").

    edit: Detiene el rebase para que cambies el código de ese commit.

    squash: (Muy útil) Mezcla ese commit con el anterior. Ideal para unir 10 commits de "arreglo error" en uno solo bien explicado.

    drop: Borra el commit por completo.

2. Git Cherry-pick (El Ladrón de Código)

El Escenario: Tienes una rama llamada pruebas-locas donde hiciste experimentos. El 90% es basura que no sirve, pero hay un commit específico donde arreglaste un error crítico del login.cpp. El ID de ese commit es a1b2c3d.
Ejemplo de git cherry-pick a1b2c3d:

    Te vas a tu rama principal: git checkout main

    "Robas" solo ese cambio: git cherry-pick a1b2c3d

    Resultado: Solo el arreglo del login aparece en main. Los otros experimentos basura se quedan en la otra rama. ¡Eficiencia pura!

3. ¿Qué hacer cuando sale mal? (Los Salvavidas)

A veces, al hacer rebase o cherry-pick, Git se detiene porque hay un Conflicto (la misma línea de código fue cambiada en dos sitios).
Ejemplo de abort vs continue:

    Escenario A (Pánico): Ves que hay 50 archivos con conflictos y no tienes tiempo.

        Comando: git rebase --abort (o git cherry-pick --abort)

        Efecto: El tiempo regresa atrás. Es como si nunca hubieras ejecutado el comando. Todo vuelve a la normalidad.

    Escenario B (Solución): Abres el archivo, ves las marcas <<<< HEAD, borras lo que no sirve y dejas el código limpio.

        git add archivo_arreglado.cpp

        git rebase --continue (o git cherry-pick --continue)

        Efecto: Git dice "¡Gracias!" y sigue con el siguiente commit.
